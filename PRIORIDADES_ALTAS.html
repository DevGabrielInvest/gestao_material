1️⃣ Paginação e Limite de Registros

Backend Frontend Esforço Médio
Problema: Atualmente, todas as rotas carregam todos os registros sem limite. Com 10k+ itens, isso causará timeout e consumirá muita memória.

📋 Implementação:

  • Backend: Adicionar parâmetros limit e offset em:
    • GET /api/inventory?limit=50&offset=0
    • GET /api/requests?limit=20&offset=0
    • GET /api/custody?limit=30&offset=0
    • GET /api/movements?limit=20&offset=0
    • GET /api/activity?limit=10&offset=0
  • Frontend: Adicionar paginação visual (próximo/anterior) ou "carregar mais"
  • Resposta: Incluir total e hasMore no JSON

2️⃣ Índices no Banco de Dados

Database Esforço Baixo
Problema: Sem índices, buscas por email, status ou categoria ficam lentas (full table scan).

📋 Implementação:

  • Criar índice simples: CREATE INDEX idx_users_email ON users(email);
  • Criar índice simples: CREATE INDEX idx_requests_status ON requests(status);
  • Criar índice simples: CREATE INDEX idx_inventory_category ON inventory(category);
  • Criar índice composto: CREATE INDEX idx_requests_user_date ON requests(requester_email, date);
  • Criar índice composto: CREATE INDEX idx_custody_inventory_status ON custody(inventory_id, status);
  • Executar npm run schema após adicionar ao migrate.js

3️⃣ Validação Robusta de Entrada

Segurança Backend Esforço Médio
Problema: APIs aceitam qualquer valor. Usuário pode enviar strings com 10MB, números negativos inválidos, tipos errados.

📋 Implementação:

  • Strings: Máximo 255 caracteres para name/code, 2000 para reason/notes
  • Números: quantity e value devem ser >= 0
  • Enums: Validar role, status, type contra lista pré-aprovada
  • Email: Validar formato com regex simples
  • Data: Validar formato YYYY-MM-DD
  • Resposta de erro: Retornar 400 com mensagem clara se inválido

4️⃣ Rate Limiting (Proteção contra Brute Force)

Segurança Backend Esforço Médio
Problema: Qualquer pessoa pode tentar senhas infinitas sem limite. Usar express-rate-limit.

📋 Implementação:

  • Instalar: npm install express-rate-limit
  • Login: 5 tentativas por minuto por IP
  • API geral: 100 requisições por minuto por token/IP
  • Exemplo: Retornar 429 (Too Many Requests) com retry-after header
  • Mensagem: "Muitas tentativas. Tente novamente em 5 minutos"

5️⃣ Tratamento de Erros Aprimorado (Frontend)

Frontend UX Esforço Médio
Problema: Erros silenciosos ou console.error. Usuário não sabe o que aconteceu.

📋 Implementação:

  • Loading state: Mostrar spinner enquanto carrega
  • Mensagens visuais: Toast ou modal com erro em vermelho
  • Retry automático: Para erros 5xx, tentar 2x após 1s
  • Validação antes: Verificar campos antes de enviar
  • Resposta clara: Ex: "Erro ao salvar. Tente novamente" em vez de código de erro
  • Offline detection: Avisar se não há conexão

6️⃣ Responsividade Mobile

Frontend UX Esforço Alto
Problema: Layout não se adapta para telas pequenas. Tabelas saem da tela, botões ficam minúsculos.

📋 Implementação:

  • Viewport: Adicionar <meta name="viewport" content="width=device-width, initial-scale=1"> no HTML
  • Media queries: Breakpoints em 768px, 1024px
  • Tabelas: Converter para cards em mobile (display: grid)
  • Navigation: Menu hambúrguer em telas pequenas
  • Botões: Mínimo 44x44px (toque confortável)
  • Fontes: Aumentar em mobile (mínimo 16px)
  • Teste: Abrir em DevTools modo mobile (F12)

7️⃣ Confirmação em Ações Destrutivas

Frontend UX Esforço Baixo
Problema: Um clique acidental em "Recusar solicitação" ou "Devolver equipamento" não pode ser desfeito.

📋 Implementação:

  • Modal de confirmação: "Tem certeza que deseja recusar esta solicitação?"
  • Botões: Cancelar (cinza) vs Confirmar (vermelho)
  • Ações a proteger:
    • Aprovar/recusar solicitação
    • Devolver equipamento
    • Deletar item de inventário
    • Deletar movimento
  • Mensagem: Explicar brevemente o que vai acontecer
